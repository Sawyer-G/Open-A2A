# 贡献指南

## 1. 协作流程

### 1.1 基本流程

```
Fork 仓库 → 创建分支 → 开发 → 提交 → 发起 Pull Request → Code Review → 合并
```

### 1.2 分支策略

参见 [git.md](./git.md)。

---

## 2. 提交规范

### 2.1 Conventional Commits 格式

```
<type>(<scope>): <subject>

[可选的 body]

[可选的 footer]
```

### 2.2 Type 类型

| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档变更 |
| `style` | 代码格式（不影响逻辑） |
| `refactor` | 重构 |
| `perf` | 性能优化 |
| `test` | 测试相关 |
| `chore` | 构建、依赖等杂项 |

### 2.3 示例

```
feat(core): 添加意图广播模块
fix(example): 修复 Merchant 订阅主题错误
docs(standards): 新增项目结构规范
```

---

## 3. Pull Request 规范

### 3.1 PR 标题

- 与提交信息格式一致
- 简洁描述变更内容

### 3.2 PR 描述模板

```markdown
## 变更类型
- [ ] 新功能
- [ ] Bug 修复
- [ ] 文档
- [ ] 其他

## 变更说明
（简要描述做了什么）

## 相关 Issue
（如有，关联 #123）

## 检查清单
- [ ] 代码符合项目规范
- [ ] 已添加/更新测试
- [ ] 文档已同步更新（如涉及中英文档，请同时更新）
```

### 3.3 Code Review 期望

- 逻辑正确、可读性良好
- 符合项目结构与命名规范
- 无敏感信息泄露
- 文档与代码一致

---

## 4. Issue 规范

### 4.1 标题

- 简洁描述问题或建议
- 可加前缀：`[Bug]`、`[Feature]`、`[Docs]`

### 4.2 内容

- 清晰描述问题或需求
- 提供复现步骤（Bug）或使用场景（Feature）
- 环境信息（如相关）

---

## 5. 沟通与礼仪

- 尊重不同观点，理性讨论
- 使用中文或英文均可，保持一致性
- 对贡献者表示感谢
