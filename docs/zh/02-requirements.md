# 产品需求

## 1. 核心功能清单 (MVP: 「一碗面条的旅程」)

| 功能模块 | 必须实现的操作 | 用户感受到的效果 |
|----------|----------------|------------------|
| **全网发现 (Discovery)** | Agent A 发布「我想吃面」的意图，网络中的 B (商家) 能感知并响应 | AI 自动找到了符合我口味的附近面馆 |
| **意图协商 (Negotiation)** | A 与 B 的 Agent 自动沟通：忌口（不加辣）、出餐时间、价格 | 我只需要说一句话，AI 帮我完成了所有的售前沟通 |
| **三方调度 (Coordination)** | B-Agent 确认订单后，自动向网络广播配送意图，C (骑手) 接单 | 商家不用手动找配送，配送员直接收到了 Agent 派发的任务 |
| **价值自动结算** | 订单交付后，Agent A 自动向 B 和 C 支付报酬（模拟） | 钱直接给到做面的人和送面的人，没有中间商扣费 |
| **标准化协议栈** | 定义一套 Agent 间公用的语义字典（如 `Order_Food`, `Logistics_Request`） | 不同的开源 Agent 只要遵循协议，就能互相「听懂」 |

## 2. 产品边界

- **中心化审核**：本项目【绝对不碰】中心化的内容审核或信用背书
- **硬件部署**：不考虑低功耗设备的适配（假设硬件资源充足）
- **交互界面**：不开发复杂的移动端 App，主要输出 **API 规范和开源协议参考实现**

## 3. 用户故事：深夜的一碗牛肉面（无平台版）

**场景**：周五深夜 23:30，程序员老张（Alice）刚加完班，感到精疲力竭。他对手机（个人 AI 舱）说了一句：「我想吃碗热面条，老规矩，不加香菜，30分钟内送到。」

### A. 意图广播与毫秒级发现 (Discovery)

- **老张的 Agent (A)** 立即被激活，从 **Solid Pod** 中读取历史口味（不吃香菜、偏好手工面、坐标、支付预算）
- **A-Agent** 向 **Open-A2A 网络** 广播加密意图包：`{Action: Food_Order, Type: Noodle, Location: [X, Y], Constraints: [No_Coriander, <30min]}`
- 方圆 3 公里内，50 多个正在运行的**商家 Agent** 瞬间接收到了这个广播

### B. 异构 Agent 的自动博弈 (Negotiation)

- **面馆 B 的 Agent** 自动查询后厨库存，发现还有最后一份手工拉面，立即回复 Offer：`18 UNIT`
- **面馆 D 的 Agent** 报价 `16 UNIT`，但 **A-Agent** 识别出 D 昨天的信誉分评价中有「面条过软」的记录，自动 pass
- **A-Agent** 秒选 B，并在 0.5 秒内完成基于 **DID** 的数字签名签约

### C. 去中心化的劳动力调度 (Logistics)

- **商家 B-Agent** 确认订单后，立刻向网络发布**配送招标**
- 骑手小王（Carrier C）的 **C-Agent** 监测到订单路径与小王回家的路重合度 95%，且报酬符合预期
- **C-Agent** 自动接单，并向老张的 **A-Agent** 发送预计到达时间  
  **整个过程，没有任何调度员参与，也没有平台抽走 20% 的配送费**

### D. 原子化结算与价值归位 (Settlement)

- 20 分钟后，小王将面送到
- 老张的手机感知到小王的物理距离（或扫码确认），触发 **Open-A2A 交付证明**
- **瞬间发生的结算**：老张支付的 `21 UNIT`（18面+3运费）被**智能合约**瞬间拆分：
  - `18 UNIT` 实时进入商家 B 的钱包（无账期，无需等待平台提现）
  - `3 UNIT` 实时进入骑手小王 C 的钱包（全额获取，无平台抽成）
- 老张吃到了面，商家多赚了钱，骑手拿到了全额运费。**这笔交易中，没有一分钱流向中间商**

## 4. 运行规则与硬性约束

### 技术栈要求

- **开源友好**：必须使用完全开源的技术栈（如 Python, Rust, libp2p）
- **架构开放**：所有通信必须基于 DID（去中心化身份），拒绝任何形式的私有账号体系

### 设计原则

- **高性能优先**：假设算力无限，但对通信延迟有极高要求，必须实现亚秒级的意图匹配
- **可扩展性**：协议必须能支持从「买面条」扩展到「买机票」、「组建临时开发团队」等复杂任务

### 待确认：关键盲点与 PM 建议

| 盲点 | PM 建议 |
|------|---------|
| **全局索引 (Global Indexing)**：在没有平台的情况下，A 怎么找到 B？ | 引入 **DPI (Decentralized Protocol Index)**，类似 P2P 网络中的分布式哈希表 (DHT) |
| **纠纷仲裁**：如果面条送到了但碎了，谁来评理？ | 预留「仲裁 Agent」接口，用户可选择信任的第三方 Agent（可能是 DAO 组织）作为多签仲裁方 |
| **垃圾消息防范**：如何防止 10000 个 B-Agent 同时骚扰 A-Agent？ | 引入基于 DID 的**反垃圾门槛（如微量质押）** |
