# 设计原则

> 本文档定义 Open-A2A 的核心设计目标与原则，是理解架构与需求的「北极星」。

---

## 1. 核心目标（非送餐）

Open-A2A 的**核心目标**不是实现送餐、外卖或任何具体业务，而是：

| 目标 | 说明 |
|------|------|
| **跨网络 Agent 通信** | 不同 Agent 可能处于各自网络（不同 NATS 集群、不同协议、不同信任域），框架需解决「如何发现彼此、如何建立通信」 |
| **数据交互的形式与逻辑** | 定义 Agent 间消息的**结构**（格式、主题、序列化）与**交互模式**（广播-收集、请求-响应、委托-确认等），使其可复用到任意协作场景 |

送餐场景只是**灵感来源**和**示例实现**，用于具象化上述问题，而非框架要达成的业务目标。

---

## 2. 设计原则

### 2.1 协议层，非业务层

- Open-A2A 定义「Agent 间如何通信」，不定义「买什么、送什么、结算什么」
- 业务语义由各 domain 自行扩展（`intent.{domain}.{action}` 中的 `domain` 可任意）

### 2.2 可扩展的交互模式

- 意图广播 + 多响应收集
- 请求 + 定向回复（reply_to）
- 子任务委托（A 委托 B，B 再委托 C）
- 这些模式应抽象为可组合的**交互原语**，而非写死在某一业务流程中

### 2.3 传输层可替换

- 当前以 NATS 为参考实现，但架构应预留**传输适配器**抽象
- 未来可支持 HTTP、WebSocket、DHT、P2P 等不同底层，以应对「各自网络」的异构性

### 2.4 身份与数据主权

- Agent 身份不依赖中心化账号，采用 DID 等去中心化标识
- 数据交互遵循最小化原则，支持可验证凭证（VC）等机制

---

## 3. 与示例场景的关系

| 概念 | 角色 |
|------|------|
| **送餐 / 外卖** | 灵感来源、Demo 示例、验证协议可行性的用例 |
| **Intent / Offer / OrderConfirm** | 通用交互模式在「订单」场景下的具象化，可抽象为 Request / Proposal / Accept |
| **Consumer / Merchant / Carrier** | 示例中的角色命名，其他场景可对应为 Requester / Provider / Delegator 等 |

阅读需求文档与架构文档时，应区分：**哪些是通用能力**，**哪些是送餐示例特有的表述**。
